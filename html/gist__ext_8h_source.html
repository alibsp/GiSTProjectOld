35 /*
36  * Interface for Gist extension classes.
37  *
38  * The procedures here are mostly node layout abstractions, only
39  * union{Key,Page}(), check(), print*() and eqQuery() are data
40  * abstractions.
41  *
42  * The subclasses of gist_ext_t are supposed to implement the node
43  * layout aspects, and offer a cleaner interface that requires the
44  * user to specify only the data abstraction routines.
45  *
46  * The subclass implementer can use any of the gist_p member functions
47  * to implement a specific node layout. Also, every page contains a BP
48  * (slot index: gist_p::bpSlot), except for the root page.
49  *
50  * The subclass implementation must guarantee that
51  *
52  * - every entry (consisting of some key data and a data/child
53  * pointer) on the page is stored as a keyrec_t (with keyrec_t as a
54  * header and the key and data following) and has a slot associated
55  * with it, no matter how much the key is compressed (the intention
56  * is that the data and child pointers are accessible via
57  * gist_p::rec(index).elem() and gist_p::rec(index).child());
58  *
59  * - slot indices are stable until the next *successful* call to
60  * insert(), remove(), updateKey() and pickSplit() (i.e., they don't
61  * change between a call to pickSplit() and remove(); also,
62  * updateKey() must not change the slot assignments if it returns
63  * with an error);
64  *
65  * The following things are *not* required:
66  *
67  * - using a contiguous block of slots in the slot array (used and
68  * unused slots may alternate)
69  *
70  * - the information to reconstruct a full key need not be stored in a
71  * single entry, as long as getKey() can reconstruct the key given
72  * the associated slot index
73  *
74  * In order to implement logical undo of data item insertion and
75  * deletion in Shore, we need to be able to associate extension
76  * objects with log records and access those extension objects at
77  * recovery time. The type gist_ext_ids records IDs for extension
78  * objects and the array gist_ext_list maps those IDs to the
79  * corresponding objects. Every new extension object should have an
80  * entry in gist_ext_ids, have its own myId set to that entry and its
81  * myName set to some descriptive name (used for amdb) as well as
82  * "register" itself in gist_ext_list. */
83

192  // Computes BP of the predicates contained on the page plus one or two
193  // extra predicates added to the page.
194  // - 'bp' must point to a buffer of size gist_p::max_tup_sz (or
195  // larger).
196  // - 'bpIsValid' indicates whether or not 'bp' contains a
197  // valid instance of the BP type, i.e., one that represents the
198  // current contents of 'page. If 'bpIsValid', only pred1/-2 need to be
199  // added to it. If '!bpIsValid', the BP must be generated from scratch
200  // for all of the predicates on the page (plus pred1/-2). The latter is required
201  // for bulk-loading (for the most recently constructed page, for which
202  // no BP exists) and for deletion (there is no "incremental
203  // deletion," i.e., we cannot (in general) apply some update
204  // operation to the old BP that is guaranteed to result in an
205  // adequately accurate BP).
206  // - 'pred1' and 'pred2' are the extra predicates added to the page (and, if
207  // the BP is valid, those predicates would not be included in 'bp'). Either
208  // one can be empty (pred.size() == 0).
209  // - 'bpChanged' indicates whether the output 'bp' is different
